# Fish-Analogous rc Module: Analysis

## What Fish Actually Does

Fish's NixOS module is sophisticated because fish is designed for it:

1. **Native system-wide config**: Fish reads `/etc/fish/config.fish` natively — no patches needed.
2. **Three-phase init in one file**: `/etc/fish/config.fish` uses `status is-login` and `status is-interactive` with guard variables (`__fish_nixos_*_config_sourced`) to run `shellInit`, `loginShellInit`, and `interactiveShellInit` exactly once each.
3. **POSIX environment integration**: Fish can't read POSIX `export` syntax, so NixOS either:
   - Translates POSIX → fish via `babelfish`
   - Runs POSIX via `foreign-env` plugin and imports vars
4. **`/etc/shells` registration**: Fish adds itself so `chsh` works.
5. **Completion generation**: Auto-generates completions from man pages of installed packages.
6. **Vendor directories**: Sources config/functions/completions from other packages.

## What a Fish-Analogous rc Would Require

rc is architecturally much simpler than fish — and much more constrained:

| Feature               | Fish                             | rc                                                                              |
| --------------------- | -------------------------------- | ------------------------------------------------------------------------------- |
| System-wide config    | `/etc/fish/config.fish` (native) | **None** — `main.c` hardcodes `$home/.rcrc`                                     |
| Interactive detection | `status is-interactive`          | None — `.rcrc` sourced with `interactive = FALSE` temporarily                   |
| Login detection       | `status is-login`                | `dashell = (*argv[0] == '-')` — only used to decide whether to read `.rcrc`     |
| POSIX syntax support  | Needs translation                | Can read some POSIX, but `export VAR=val` fails (rc: `export` is for functions) |
| Config phases         | 3 phases in 1 file               | Only 1 file, 1 phase                                                            |
| `/etc/shells`         | Yes                              | **Not yet implemented**                                                         |

## The Fundamental Gap: No `/etc/rcrc`

The biggest blocker is that rc reads **only** `$home/.rcrc`, with no env override:

```c
// main.c:108
rcrc = concat(varlookup("home"), word("/.rcrc", NULL))->w;
fd = rc_open(rcrc, rFrom);
```

For a fish-analogous rc module, we'd need to patch rc to read `/etc/rcrc` first:

```c
// Hypothetical patch
char *sysrc = "/etc/rcrc";
int sysfd = rc_open(sysrc, rFrom);
if (sysfd != -1) {
    pushfd(sysfd);
    bool was_interactive = interactive;
    interactive = FALSE;
    doit(TRUE);
    interactive = was_interactive;
    close(sysfd);
}
// Then existing ~/.rcrc code...
```

This is ~10 lines, but it's a **structural change** to a shell that hasn't changed its init model in 30+ years.

## What Would a Patched rc Module Look Like?

If we patched rc, the NixOS module could generate `/etc/rcrc`:

```rc
# /etc/rcrc: DO NOT EDIT -- generated by NixOS

# General init (always run for login shells)
if (! ~ $#__rc_general_sourced 0) {}
if not {
    __rc_general_sourced = 1
    # NixOS environment in rc syntax...
    # shellInit...
}

# Login init
if (! ~ $#__rc_login_sourced 0) {}
if not {
    __rc_login_sourced = 1
    # loginShellInit...
}

# Interactive init (best-effort — rc sets interactive=FALSE during .rcrc)
if (! ~ $#__rc_interactive_sourced 0) {}
if not {
    __rc_interactive_sourced = 1
    # prompt, history, aliases...
}

# Source user config
if (test -r $home/.rcrc) { . $home/.rcrc }
```

And the home-manager module would generate `~/.rcrc` with user-specific config.

## Practical Paths Forward

### Path A: Patch rc (most fish-like)

- Add `/etc/rcrc` support to rc
- Generate `/etc/rcrc` on NixOS/darwin
- Register in `/etc/shells`
- **Pros**: Full system integration, proper login shell support
- **Cons**: Patch maintenance, upstream unlikely to accept

### Path B: Home-manager only (current)

- `~/.rcrc` handles everything
- NixOS/darwin just install the package
- **Pros**: Zero patches, minimal maintenance
- **Cons**: No system-wide config, broken login shell on NixOS (no PATH, etc.)

### Path C: Wrapper script

- Create `/run/current-system/sw/bin/rc-login`:
  ```sh
  #!/bin/sh
  . /etc/profile  # Get NixOS environment
  exec /nix/store/.../bin/rc "$@"
  ```
- Register wrapper in `/etc/shells`
- **Pros**: No rc patch needed
- **Cons**: Hacky, `ps` shows wrapper not rc, breaks `$0` detection

### Path D: Environment-only NixOS module

- NixOS module sets env vars via `environment.variables` (which rc inherits)
- No init file generation on NixOS
- **Pros**: Simple, no patches
- **Cons**: Still no system-wide rc config, aliases, etc.

## Recommendation

For rc specifically, the cost/benefit of Path A (patching) is questionable:

- rc's userbase is tiny and technical — they can manage `~/.rcrc`
- rc's philosophy is minimalism — adding `/etc/rcrc` is antithetical
- The current home-manager-only approach (Path B) works for the 99% use case

If Lane wants rc as a first-class NixOS login shell, **Path A** is the right choice. Otherwise, the current implementation is the pragmatic one.
